# Техническая спецификация

Проект: ИИ-ассистент для управления личными задачами из WhatsApp, email и календаря
Версия: 1.0
Дата: 2026-04-15

## 1. Цель и границы проекта

Система должна стать единым центром фиксации и исполнения задач из трех каналов:
- WhatsApp
- email (через пересылку на выделенный адрес)
- календарь (занятость и контекст)

Система ориентирована на практическое ежедневное использование, а не на чат-диалоги ради диалогов.

## 2. Ключевая ценность (Business Outcome)

Снижение риска пропуска дедлайнов, встреч и обязательств за счет:
- автоматического извлечения задач из входящих сообщений и писем;
- унифицированного хранилища задач и контекста;
- ежедневной и недельной повестки с учетом календарной занятости;
- proactive-напоминаний по критичным/ближайшим/просроченным задачам.

## 3. Основные пользовательские роли

- Владелец аккаунта (основной пользователь): создает и управляет задачами.
- Система-ассистент: извлекает, уточняет, планирует, напоминает.
- Интеграции (WhatsApp/email/calendar): источники событий.

## 4. Функциональный объем (Scope)

### 4.1 Управление задачами

Поддерживаются операции:
- create: создание задачи
- update: редактирование описания
- reschedule: перенос срока/времени
- reprioritize: изменение приоритета
- complete: завершение
- cancel: отмена
- delete: удаление
- tag: назначение тегов/категорий
- link_source: привязка источника (сообщение, письмо, календарное событие)

### 4.2 Каналы ввода

1. WhatsApp:
- прием произвольного текста;
- извлечение intent и параметров;
- подтверждение операции.

2. Email:
- прием писем на выделенный адрес;
- разбор комментария пользователя и цитируемого треда;
- выделение задачи/дедлайна/контекста;
- сохранение metadata источника.

3. Календарь:
- чтение занятости и событий;
- определение свободных слотов;
- связывание задач с встречами (follow-up).

### 4.3 Повестки

1. Повестка на день:
- встречи на сегодня (хронологически)
- задачи с дедлайном сегодня
- просроченные задачи
- follow-up задачи после встреч
- рекомендации на свободные окна
- блок "Критично сегодня"

2. Повестка на неделю:
- ключевые встречи по дням
- дедлайны задач на неделе
- просроченные
- high-priority без точного срока
- предупреждения о перегрузке
- рекомендации что сделать заранее

### 4.4 Напоминания

Типы напоминаний:
- точечные (в указанное время)
- за N минут/часов до дедлайна
- утренний приоритетный дайджест
- вечерний список незавершенных
- отдельный alert по просроченным

### 4.5 Уточнения (Clarification)

Если не хватает данных (срок/тип записи/исполнитель), система задает короткий вопрос.
Если вопрос не критичен, задача сохраняется с флагом low_confidence.

## 5. Нефункциональные требования (NFR)

- Timezone-aware: интерпретация дат строго в часовом поясе пользователя.
- Надежность: гарантированная обработка входящих событий (at-least-once + idempotency).
- Безопасность: шифрование данных at-rest и in-transit.
- Наблюдаемость: структурные логи, метрики, трассировка.
- Расширяемость: добавление новых каналов без переработки ядра.
- SLA (рекомендуемо для MVP):
  - подтверждение входящего сообщения: до 5 секунд;
  - формирование повестки дня: до 3 секунд (p95);
  - напоминания: отклонение не более 1 минуты.

## 6. Архитектура решения

### 6.1 Логическая схема

1. Ingestion Layer:
- WhatsApp Webhook Receiver
- Email Inbound Processor
- Calendar Sync Worker

2. Understanding Layer:
- NLP Parser (rules + LLM extraction)
- Normalizer (даты, приоритеты, действия)
- Confidence Scorer

3. Task Core:
- Task Service
- Reminder Service
- Agenda Service
- Linking Service (задача <-> событие)

4. Delivery Layer:
- WhatsApp Outbound Sender
- Email Outbound Sender (опционально)
- Notification Scheduler

5. Persistence Layer:
- OLTP DB (PostgreSQL)
- Queue (RabbitMQ/SQS/Kafka)
- Object Storage (сырой email/attachments metadata)

### 6.2 Рекомендуемый стек (MVP)

- Backend: Python (FastAPI) или Node.js (NestJS)
- DB: PostgreSQL
- Queue/Scheduler: Redis + Celery/Sidekiq/BullMQ
- NLP: LLM API + правила для дат/приоритетов
- Интеграции:
  - WhatsApp Business API (Meta Cloud API)
  - Email: inbound mailbox + IMAP/API провайдера
  - Calendar: Google Calendar API и/или Microsoft Graph

## 8. NLP/извлечение: правила и pipeline

1. Предобработка:
- очистка цитат и сигнатур email
- языковая нормализация (ru/en)
- выделение потенциальных action phrases

2. Извлечение сущностей:
- action (что сделать)
- due_date / due_time
- priority markers
- relation-to-meeting (после встречи, до звонка и т.д.)

3. Нормализация времени:
- относительные сроки в абсолютный timestamp
- timezone пользователя обязателен

4. Оценка уверенности:
- high: срок и действие однозначны
- medium: действие ясно, срок относительный/широкий
- low: неясная формулировка, требуется вопрос

5. Политика уточнений:
- если не определено действие -> спросить обязательно
- если не определен срок и есть приоритет critical/high -> спросить
- если срок неоднозначен (например, "в пятницу" при переходе недели) -> уточнить

## 9. Алгоритм повестки на день

Вход: пользователь, дата D

1. Получить встречи D.
2. Получить задачи:
- due_at в D
- overdue open tasks
- follow-up по встречам D
- high/critical без due_at
3. Рассчитать свободные окна между встречами.
4. Кандидаты в окна:
- short tasks (оценка <= 30-60 минут)
- high-priority first
5. Сформировать блоки ответа:
- Встречи
- Критично сегодня
- Дедлайны сегодня
- Просрочено
- Рекомендуемые окна

## 10. Алгоритм повестки на неделю

Вход: пользователь, интервал W (пн-вс)

1. Встречи по дням.
2. Дедлайны задач по дням.
3. Просроченные и high-priority без срока.
4. Оценка перегрузки:
- день считается перегруженным, если:
  - занято календарем > 70%
  - и/или дедлайнов > заданного порога
5. Рекомендации переноса/раннего выполнения.

## 13. Обработка ошибок и граничные случаи

- Дубли входящих событий: idempotency key по message_id/email_message_id.
- Нет доступа к календарю: повестка строится без слотов, с явным предупреждением.
- Пустое/неинформативное сообщение: сохранить как заметку или запросить уточнение.
- Конфликт времени: предложить альтернативный свободный слот.

## 14. Безопасность и комплаенс

- OAuth2 для календаря и email-интеграций.
- Подпись вебхуков (проверка source authenticity).
- Шифрование PII/конфиденциальных полей.
- Ротация ключей и аудит-логи действий.
- Политика хранения: архивирование/удаление по сроку.

## 15. MVP-план (8-10 недель)

Этап 1 (Недели 1-2):
- Базовый backend, БД, auth, сущности задач.

Этап 2 (Недели 3-4):
- WhatsApp inbound/outbound, создание задач, подтверждения.

Этап 3 (Недели 5-6):
- Email ingestion, парсинг тредов, сохранение source links.

Этап 4 (Недели 7-8):
- Calendar sync, day agenda, reminder scheduler.

Этап 5 (Недели 9-10):
- Week agenda, overload detection, hardening и observability.

## 16. Критерии приемки (Acceptance Criteria)

1. Создание задачи из WhatsApp:
- Given пользователь отправил свободный текст
- When обработка завершена
- Then задача сохранена и отправлено подтверждение

2. Создание задачи из email:
- Given переслано письмо с поручением
- When письмо обработано
- Then задача создана, источник сохранен, дедлайн распознан или запрошено уточнение

3. Повестка на день:
- Given есть встречи и задачи
- When пользователь запрашивает повестку
- Then ответ содержит встречи, критичное, дедлайны, просроченное и рекомендации по окнам

4. Напоминания:
- Given для задачи выставлено время/дедлайн
- When наступает remind_at
- Then пользователь получает уведомление по выбранному каналу

5. Уточнения:
- Given низкая уверенность парсинга
- When ключевые атрибуты не распознаны
- Then ассистент задает короткий уточняющий вопрос

## 17. Риски и меры

- Риск: неточный NLP-парсинг.
  - Мера: confidence scoring + clarification + лог ошибок.
- Риск: блокировки интеграций/лимиты API.
  - Мера: retry policy, backoff, dead-letter queue.
- Риск: перегруз уведомлениями.
  - Мера: quiet hours и policy-ограничения.

## 18. Что не входит в MVP

- Мультипользовательские команды и делегирование между сотрудниками.
- Полноценный web/mobile UI (кроме служебной админ-панели).
- Автоматическое выполнение действий в сторонних системах (RPA).

## 19. Минимальные метрики успеха (после запуска MVP)

- >= 85% входящих сообщений корректно классифицируются как задача/заметка.
- >= 80% задач имеют корректно распознанный срок (или закрываются уточнением).
- >= 95% напоминаний доставлены в SLA.
- Снижение доли просроченных задач у пользователя не менее чем на 20% за 6-8 недель.
